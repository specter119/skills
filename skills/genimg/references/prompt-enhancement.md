# Prompt Enhancement Guide

## Rationale
Gemini image models are better at understanding rich, narrative descriptions; simple keywords often fall short of optimal results. Use a 50–150 word paragraph that weaves together subject, action, setting, lighting, composition, and style into a small story.

## Key Elements
| Element | Description | Example |
|------|------|------|
| Subject | Who or what the main focus is | "A robot barista with blue optical lenses" |
| Action | What the subject is doing | "Brewing coffee in a futuristic café" |
| Setting | The surrounding environment | "A futuristic café" |
| Lighting | Nature and direction of light | "Soft ambient light cast from overhead neon tubes" |
| Technical | Lens / quality / depth of field, etc. | "Shallow depth of field, 4K quality" |
| Style | Overall tone matching `-s` | "Modern tech aesthetic" |

## Auto-Supplement Rules
- If text is not explicitly requested, append `-n "text, words, letters, labels"`.
- Add style-matching descriptors based on the `-s` parameter.
- Add appropriate lighting and composition suggestions based on scene type.

## Example
User input:
```plain
Draw a concept diagram for the A2A protocol with a tech feel
```

Enhanced:
```plain
An isometric 3D illustration depicting the A2A protocol concept. Multiple AI agents represented as glowing geometric nodes connected by flowing data streams. Clean, modern tech aesthetic with blue accent lighting. Soft diffused top-down lighting creates depth. Professional, corporate style suitable for technical presentation. Sharp focus, high quality rendering.
```

## Edit Mode Prompting
When modifying an existing image, expand the user's edit instruction to:
```plain
Adjust the lighting to warmer tones. Add golden hour warmth with soft orange highlights. Maintain the original composition and subject placement. Keep all other elements unchanged.
```

## Execution Tips
- Use Flash model for fast exploration first, then Pro model for refinement.
- Use `--variants` + `--grid` to auto-generate a multi-image grid for comparison. See `references/cli-workflow.md` for detailed CLI operations.

## Visual Anti-Defaults

AI image models have their own "default aesthetic." The following patterns are not forbidden, but should not be produced unconsciously as the default:

### Avoid Default Output

- **Blue-purple tech gradient backgrounds** — models favor indigo→cyan and purple→blue "trustworthy" gradients. If the user has not requested a gradient, default to solid color or natural lighting.
- **Floating abstract geometry** — glowing cubes, spheres, and hexagonal grids carry no semantic content. If a concept image needs shapes, make the shapes carry meaning.
- **Fake UI / fake dashboards** — interface elements generated by models typically have unreadable text and illogical layouts. When a UI screenshot is needed, use a real screenshot or code-generated output.
- **Excessive glow and flare** — lens flare, neon glow, light particles. Use sparingly for sci-fi styles, but do not let them become the default effect on every image.

### Reference Negative Prompts

When the above default tendencies are detected, append a negative prompt:

```plain
-n "generic gradient background, floating abstract shapes, fake UI elements, lens flare, glowing particles"
```

Note: If the user explicitly requests a particular style (e.g., "sci-fi glowing nodes"), respect the user's intent and do not add that type of negative prompt.

### Evaluation Criterion

Ask yourself: if this visual element were removed, would the core message of the image change? If not, the element is most likely the model's default filler rather than a deliberate design choice.
